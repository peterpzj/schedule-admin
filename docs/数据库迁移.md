# 数据库迁移

> 适用于 schedule-admin 后端（SQLite）

## 文件结构

```
backend/db/
  index.js          # 入口（getDb / initDb）
  migrate.js        # 迁移 runner
  migrations/
    001_init.sql                    # 初始 schema（与历史内嵌 SQL 等价）
    002_add_user_last_login_at_index.sql
    003_xxx.sql
```

## 工作流

```bash
# 1) 看当前状态
npm run migrate:status

# 2) 应用所有 pending（启动时自动跑，也可手动）
npm run migrate

# 3) 新建一个迁移（自动命名 NNN_name.sql）
npm run migrate:create add_patient_limit
# 生成 db/migrations/003_add_patient_limit.sql
# 编辑该文件填 DDL/DML（带可选的 Down: 段）

# 4) 应用新迁移
npm run migrate
```

## 迁移文件格式

```sql
-- Migration: add_patient_limit
-- Created: 2026-06-08

-- 在此写 DDL/DML
ALTER TABLE schedules ADD COLUMN patient_limit INTEGER DEFAULT 30;

-- Down: （可选，runner 用 --down 调用）
ALTER TABLE schedules DROP COLUMN patient_limit;
```

- 整个文件在事务中执行（all-or-nothing）
- 文件名必须是 `NNN_*.sql`（NNN = 3 位数字）
- 用 `-- Down:` 单独一行标记回滚段
- 一个文件只做一件事，复杂改动分多文件

## 跟踪表

每次成功应用会写入 `schema_migrations`：

| 列 | 含义 |
|----|------|
| version | 主键，如 `001_init` |
| applied_at | 应用时间（localtime） |
| checksum | 文件 SHA-256（防止事后修改） |
| execution_ms | 执行耗时（毫秒） |

`--status` 校验 checksum，会标记 `MISMATCH`（说明有人手动改过迁移文件）。

## 何时新建迁移

任何 DDL/DML 改动都要新建一个迁移，包括：

- 加表 / 改列 / 删列
- 加索引 / 删索引
- 数据修复（UPDATE/DELETE 限定到具体场景）

不允许：

- 直接编辑已经应用过的迁移文件（会被 checksum 校验发现）
- 在 `db/index.js` 内嵌新 SQL

## 回滚

仅当迁移文件含 `-- Down:` 段才支持回滚：

```bash
node db/migrate.js --down 002_add_user_last_login_at_index
```

回滚顺序：从最新版本往下逐个回滚（不要跳号）。

## 部署

`bootstrap.sh` / `deploy.sh` 不需要额外步骤：服务启动时 `initDb()` 自动调用 `migrate.migrateAllSync()`，未执行的迁移会被应用。

部署后建议：

```bash
cd /opt/schedule-admin/backend
npm run migrate:status
```

确认所有迁移已应用。

## 与 P0-4 备份的协作

新增迁移前**先做一次完整备份**（`node scripts/backup.js --verify`）。

如果迁移失败：

1. 服务启动报错（`migration failed, see logs above`）
2. 看日志：哪条迁移失败
3. 修复迁移文件，重新部署（runner 会从失败处继续）
4. 严重时 `--down` 回滚到上一个成功版本

## 多环境

- dev / staging / prod 共用同一套迁移
- 迁移只增不删
- 跨环境顺序由文件名（NNN 序号）保证
- 不要在环境 A 应用 003，但环境 B 直接跳到 004

## 已存在的特殊处理

### 旧库的"无迁移历史"

如果你的数据库是 2026-06 之前部署的（用了内嵌 SQL），首次升级时：

1. 启动服务 → `initDb()` 检测到 `schema_migrations` 表为空
2. 自动应用 001_init.sql
3. 因为 `IF NOT EXISTS`，所有已有表不会被破坏
4. 之后 002、003... 正常递增

无需手工操作。
