# 监控告警

> 适用于 schedule-admin 后端

## 指标端点

- `GET /metrics` — Prometheus 文本格式
- 默认开放在 `:3000/metrics`
- 不需要鉴权（与 `/api/health` 同级，绕过限流）
- 不被 metrics 自身统计（避免递归）

## 指标清单

### 进程级
- `process_uptime_seconds` — 进程启动时长
- `process_resident_memory_bytes` — RSS 内存
- `process_heap_used_bytes` / `process_heap_total_bytes` — V8 堆内存
- `process_cpu_user_seconds_total` / `process_cpu_system_seconds_total` — CPU 累计

### HTTP
- `http_requests_total{method,route,status}` — 各路由 QPS / 状态码分布
- `http_request_duration_ms{method,route}` — 响应时间 histogram（桶 5/10/25/50/100/250/500/1000/2500/5000/10000 ms）

### 业务
- `db_connected` — SQLite 连通（0/1）
- `db_writable` — SQLite 写权限（0/1）
- `db_size_bytes` — 数据库文件大小
- `service_uptime_seconds` — 服务启动时长
- `rate_limit_buckets{name=...}` — 限流器当前桶数

## 告警规则

| 规则 | 阈值（默认） | 严重度 |
|-----|-------------|--------|
| API 错误率过高 | 5xx 占比 > 5%（且请求数 > 50） | critical |
| API 平均延迟超阈值 | > 2000ms（且请求数 > 100） | warning |
| 数据库连接失败 | db_connected=0 | critical |
| 进程内存偏高 | RSS > 500MB | warning |

阈值通过环境变量调整（见下方配置）。

## 安装

```bash
# 1) 复制 .env 模板
cp .env.example .env
nano .env   # 配置 ALERT_WEBHOOK_URL 等

# 2) 启动服务（确保 /metrics 端点可达）
docker compose up -d

# 3) 安装告警 cron
sudo bash scripts/setup-alert-cron.sh

# 4) 立即测试
cd backend
node scripts/alert-check.js
```

## 配置（环境变量）

| 变量 | 默认值 | 含义 |
|-----|--------|------|
| `ALERT_WEBHOOK_URL` | 空 | 告警发送目标（企业微信/钉钉/Slack 通用 webhook；空 = 仅写日志） |
| `METRICS_URL` | `http://localhost:3000/metrics` | 指标抓取地址 |
| `ALERT_ERROR_RATE_THRESHOLD` | `0.05` | 错误率阈值（0-1） |
| `ALERT_P99_LATENCY_MS` | `2000` | 延迟阈值 ms |
| `ALERT_DISK_USAGE_PCT` | `80` | 磁盘使用率阈值 |
| `ALERT_DEDUPE_MINUTES` | `10` | 同类告警去重窗口（避免刷屏） |
| `ALERT_STATE_FILE` | `./data/.alert-state.json` | 去重状态文件 |

## 接入 Prometheus / Grafana

`/metrics` 端点遵循 Prometheus 文本格式，可直接被 Prometheus 抓取。

`prometheus.yml` 示例：

```yaml
scrape_configs:
  - job_name: schedule-admin
    metrics_path: /metrics
    static_configs:
      - targets: ['api.your-domain.com']
    scrape_interval: 30s
```

Grafana 推荐面板：
- 错误率（`rate(http_requests_total{status=~"5.."}[5m]) / rate(http_requests_total[5m])`）
- P95 延迟（`histogram_quantile(0.95, rate(http_request_duration_ms_bucket[5m]))`）
- 内存 / CPU 趋势
- 数据库大小
- 限流桶数（看是否被攻击）

## 退出码

- `0` — 正常
- `1` — /metrics 抓取失败
- `2` — 触发了告警

## 排查流程

1. `curl http://localhost:3000/metrics` 看原始数据
2. `tail -f /var/log/schedule-alert.log` 看历史告警
3. `docker compose logs backend` 看应用日志
4. `node backend/scripts/alert-check.js` 手动跑一次
