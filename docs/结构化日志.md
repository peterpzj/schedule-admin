# 结构化日志

> 适用于 schedule-admin 后端

## 概览

- 输出：JSON 到 stdout / stderr（容器友好，Docker 自动采集）
- 请求链路：每个请求生成 `requestId`，贯穿全链路
- 慢请求：默认 2000ms 阈值，超过会单独 warn
- 健康检查/指标端点不打日志（避免噪音）

## 核心模块

- `backend/lib/logger.js` - 结构化日志器（无第三方依赖）
- `backend/middleware/requestId.js` - Request ID 中间件
- `backend/middleware/httpLogger.js` - HTTP 请求日志中间件（替代 morgan）
- `backend/middleware/audit.js` - 操作日志（同时写 DB + 结构化日志）

## 使用方法

### 业务代码

```js
const log = require('./lib/logger');

log.info('user.login', { userId: 1, ip: '1.2.3.4' });
log.warn('slow.query', { durationMs: 1500, sql: 'SELECT ...' });
log.error('payment.failed', { orderId: 'o-123' }, err);

// 绑定上下文的子 logger
const reqLog = log.child({ requestId: 'r-abc', userId: 42 });
reqLog.info('schedule.create', { scheduleId: 99 });
```

### Express 路由

`req.log` 已由 requestId 中间件注入：

```js
app.post('/api/schedules', (req, res) => {
  req.log.info('schedule.create.start', { campus: req.body.campus_code });
  // ... 业务逻辑
});
```

## 输出示例

JSON 格式（生产）：

```json
{"ts":"2025-12-08T10:23:45.123Z","level":"info","msg":"http.request","service":"schedule-admin-api","requestId":"r-9a3b2c","method":"POST","route":"/api/schedules","status":201,"durationMs":45.2,"ip":"1.2.3.4","userId":42}
```

Pretty 格式（开发）：

```
2025-12-08T10:23:45.123Z INFO  [service=schedule-admin-api requestId=r-9a3b2c] http.request
```

## 字段约定

| 字段 | 含义 |
|-----|------|
| `ts` | ISO 8601 时间戳 |
| `level` | debug / info / warn / error |
| `msg` | 事件名（点分命名，如 `http.request`、`audit`） |
| `requestId` | 请求链路 ID（每个 HTTP 请求唯一） |
| `service` | 服务名（恒为 `schedule-admin-api`） |
| `method` / `route` / `status` / `durationMs` | HTTP 请求相关 |
| `userId` | 当前用户 ID（鉴权后可用） |
| `ip` | 客户端 IP（已通过 trust proxy） |
| `err` | 错误对象（自动序列化 name/message/stack） |

## 级别选择

| 级别 | 何时用 |
|-----|--------|
| `debug` | 调试用，默认关闭 |
| `info` | 正常事件（HTTP 2xx、audit 写入、启动完成） |
| `warn` | 客户端错误（HTTP 4xx）、慢请求、可恢复异常 |
| `error` | 服务端错误（HTTP 5xx）、关键操作失败、启动失败 |

## 环境变量

| 变量 | 默认值 | 含义 |
|-----|--------|------|
| `LOG_LEVEL` | `info` | 最低输出级别 |
| `LOG_PRETTY` | `false` | true = 人类可读格式 |
| `HTTP_SLOW_THRESHOLD_MS` | `2000` | HTTP 慢请求阈值 |

## 接入 ELK / Loki

### Loki（推荐）

用 [promtail](https://grafana.com/docs/loki/latest/clients/promtail/) 直接采集容器 stdout，JSON 直接解析：

```yaml
scrape_configs:
  - job_name: schedule-admin
    static_configs:
      - targets: [localhost]
        labels:
          job: schedule-admin
          __path__: /var/lib/docker/containers/*/*.log
    pipeline_stages:
      - docker: {}
      - json:
          expressions:
            level: level
            requestId: requestId
            userId: userId
            msg: msg
      - labels:
          level:
          requestId:
```

### ELK

用 Filebeat 采集 JSON 日志：

```yaml
filebeat.inputs:
  - type: container
    paths:
      - /var/lib/docker/containers/*/*.log
    json.keys_under_root: true
    json.add_error_key: true
```

## 排查流程

```bash
# 按 requestId 拉所有相关日志
docker compose logs backend | grep 'r-9a3b2c'

# 按用户查
docker compose logs backend | grep '"userId":42'

# 按错误类型
docker compose logs backend | grep '"msg":"audit.write.failed"'
```

## 与 P0-3 告警的关联

P0-3 的 `alert-check.js` 不读 stdout 日志，而是直接 scrape `/metrics` 端点。日志系统主要服务于：
- 事后追溯（保留 30+ 天）
- 排障（非告警场景的细节查询）
- 业务分析（用户行为、慢接口分布）
