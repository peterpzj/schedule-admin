# 数据库备份与恢复

> 适用于 schedule-admin 后端（SQLite + WAL）

## 文件清单

- `backend/scripts/backup.js` — 备份主脚本
- `backend/scripts/verify-backup.js` — 备份完整性校验
- `scripts/setup-backup-cron.sh` — 安装备份 cron

## 备份策略：分层保留

| 层级 | 默认保留 | 说明 |
|-----|---------|------|
| 每日 | 最近 7 份 | 每天凌晨 2 点产生一份 |
| 每周 | 最近 4 份 | 取每周日（ISO 周）的最新一份 |
| 每月 | 最近 12 份 | 取每月 1 号的最新一份 |
| 兜底 | 14 天 | BACKUP_KEEP_DAYS，强制保留所有 14 天内的备份 |

同一份备份可能同时被多个层级选中（按去重计算）。

## 备份内容

每次备份生成两个文件：

```
data/backups/
  schedule-20251208-020000.db              # 数据库快照
  schedule-20251208-020000.db.meta.json    # 元数据（哈希、字节数、表数）
```

元数据示例：

```json
{
  "filename": "schedule-20251208-020000.db",
  "createdAt": "2025-12-08T02:00:00.000Z",
  "dbSizeBytes": 524288,
  "sha256": "a1b2c3d4...",
  "tableCount": 11,
  "requiredTablesOk": true
}
```

## 安装

```bash
# 1) 创建数据目录
mkdir -p backend/data/backups
mkdir -p logs

# 2) 立即手动跑一次
cd backend
node scripts/backup.js --verify

# 3) 安装 cron（每天凌晨 2 点）
sudo bash scripts/setup-backup-cron.sh
```

## 常用命令

```bash
# 列出所有备份
ls -lht backend/data/backups/

# 看最近一份的元数据
cat backend/data/backups/schedule-*.meta.json | tail -1

# 校验所有备份
node backend/scripts/verify-backup.js

# 校验最近一份
node backend/scripts/verify-backup.js --latest

# 校验指定文件
node backend/scripts/verify-backup.js --file=schedule-20251208-020000.db

# JSON 输出（便于监控接入）
node backend/scripts/verify-backup.js --json | jq '.summary'
```

## 恢复流程

### 场景一：最近一次部署出错，想回滚 DB

```bash
# 1) 找一份健康备份
node backend/scripts/verify-backup.js --latest

# 2) 暂停服务
cd /opt/schedule-admin
docker compose stop backend

# 3) 恢复
cd backend
node scripts/backup.js --restore=../data/backups/schedule-20251208-020000.db

# 4) 启动服务
cd ..
docker compose up -d

# 5) 验证
curl https://api.your-domain.com/api/health
```

### 场景二：发现数据被误删，但备份是 24h 前的

1. **立即**停止服务，避免覆盖
2. 复制当前 `data/schedule.db` 到安全位置
3. 评估是否可以通过 audit_logs 表追溯删除操作
4. 从备份恢复 + 手工补录缺失数据

### 场景三：服务器整机故障

1. 在新服务器上重新部署
2. 把 `data/backups/` 目录的最新 7 份 scp 过来
3. 选最新一份 `--restore=`
4. 启动后跑 `verify-backup.js` 确认

## 关键设计

- **不阻塞写入**：用 `better-sqlite3 .backup()` 在事务一致性快照下做，不影响线上写
- **原子性**：快照过程中即使有写入，备份也是一致的
- **校验独立**：verify 脚本单独运行，可接入监控告警
- **恢复有兜底**：恢复前自动备份当前 DB 到 `*.pre-restore`
- **分层保留**：长期保留+短期高频的平衡

## 监控接入

把 `verify-backup.js --json` 接入监控告警系统（P0-3 已实现 alert-check.js 的框架）：

- 输出 `summary.fail > 0` 时告警
- 备份缺失（24h 内无新文件）时告警
- 单次备份 > 100MB 时告警（可能数据异常膨胀）

## 远程备份（可选）

当前默认本地保留。如需上传到对象存储（腾讯云 COS / 阿里云 OSS / S3）：

1. 安装对应 SDK：`npm install cos-nodejs-sdk-v5`
2. 在 `backup.js` 末尾增加 `uploadToCos(file)` 调用
3. 配置环境变量：`TENCENT_COS_BUCKET`、`TENCENT_COS_REGION`、`TENCENT_COS_SECRET_ID`、`TENCENT_COS_SECRET_KEY`

> 此项未在 P0 实施，预留为 P1。

## 容量估算

- 当前 schema 单份约 200KB ~ 2MB（取决于排班数据量）
- 按 7d + 4w + 12m 共 23 份去重后约 ~50MB
- 服务器预留 500MB 已绰绰有余
